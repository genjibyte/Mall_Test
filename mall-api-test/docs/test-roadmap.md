# 后续测试路线图（不同链路 · 去重驱动）

> 三条选链原则：**① 覆盖不同服务的样板链路**（不在同一服务/同一模式上重复）；**② 覆盖多类真实风险**（每条对准一类尚未验证的风险）；**③ 每个用例都有独立讲述价值**（与已覆盖点相同则不做）。
> 配套：[order-chain-exemplar.md](order-chain-exemplar.md)（样板范式）、[test-conventions.md](test-conventions.md)（写法）、[test-coverage.md](test-coverage.md)（现状 60 用例）。

## 1. 去重基线（已覆盖，**不再重复**）

| 已覆盖能力 | 用例 | → 后续**不要**再做的"相同点" |
|---|---|---|
| 下单金额/库存/状态/促销/券/积分/边界 | Order* (12 类) | 别再用"另一个商品/另一个券"重复算金额、扣库存、状态流转 |
| 并发超卖(库存原子性) | R4 探针 | 别再做"通用订单超卖"——只做**新维度**的限量并发(券/秒杀) |
| 鉴权**执行**(401/跨账号/403) | AuthGuard | 别再测 401/403 本身——只做**配置侧**(授权变更后动态生效) |
| 缺陷类:幂等(R1)/越权支付(R2)/回滚(R6)/泄漏(R8) | 探针 | 别再在订单上重复这些——换**新资源/新动作**验证同类风险 |
| 购物车/收货地址/领券/退货状态机/浏览/搜索/admin商品·订单管理 | 对应用例 | 这些 CRUD/状态点**已闭环**,不重复 |

## 2. 风险分类 × 覆盖（看"还差哪类风险"，而非"还差哪个接口"）

| 真实风险类别 | 现状 | 后续目标链路 |
|---|---|---|
| 计算正确性(金额/折扣) | ✅ 订单 | —（已充分，不重复）|
| 并发/限量原子性 | ✅ 库存超卖 | ○ **秒杀限量**、**券限量领取**（新维度）|
| 状态机/**时间窗口** | ✅ 订单/退货状态 | ○ 券 enable/end 时间、秒杀场次窗口（**时间**是新风险）|
| 鉴权**配置**与动态生效 | ⚠️ 仅测执行 | ○ **RBAC 配置链路**(授权→Redis 刷新→403↔200) |
| 幂等/防重 | ✅ 支付 | ○ **重复注册/重复领券/重复提交**（新动作）|
| 最终一致/传播 | ✅ MQ/ES | ○ **admin 改价上下架 → 前台/搜索传播**一致性 |
| 验证码/账户安全 | ❌ 未覆盖 | ○ **会员注册/改密**(短信码 Redis 流程) |
| 退款/积分**会计** | ⚠️ 仅 R6 探针 | ○ **退款金额与积分回退会计** |
| NoSQL(Mongo)一致性 | ❌ 未覆盖 | ○ **会员社交**(关注/收藏/足迹) |
| 输入校验/安全 | ❌ 未覆盖 | 横切:每条负例补越界/非法/注入字符 |

## 3. 后续样板链路提案（按价值排序；每条=不同服务+不同风险+独立讲述）

### P1 · 秒杀/限时购（mall-admin SmsFlashPromotion + portal）
- **服务/锚点**：`SmsFlashPromotionController` + `Session` + `ProductRelation`（admin 配场次/选品/限量）；portal 经 `HomeController` 闪购列表 + 下单走 flash 价。
- **真实风险**：**限量并发 + 每人限购 + 场次时间窗 + 秒杀价**——与 R4"通用超卖"**不同**(多了 per-member 限购、时间窗、专属价)。
- **独立讲述**：高并发抢购下"总量不超发 + 单人不超限 + 窗口外不可买 + 价取秒杀价"。
- **预估缺陷面**：限购校验竞态、窗口边界、秒杀价与原价/促销叠加。
- **去重**：不重复普通下单金额/库存；只测秒杀**专属**不变量。

### P1 · 会员注册与认证安全（mall-portal UmsMember）
- **服务/锚点**：`UmsMemberController` `/sso/getAuthCode`(写 Redis 验证码) → `/sso/register` → `/sso/updatePassword`。
- **真实风险**：**验证码流程(Redis 态) + 注册防重 + 改密**——全新风险类，现仅测过"已有账号登录"。
- **独立讲述**：取码→注册成功；**错/过期码拒绝**、**重复用户名拒绝**、改密后旧密失效/新密可登。
- **预估缺陷面**：验证码不过期/可重用、注册竞态重复、弱校验。
- **去重**：登录 happy + 401 已覆盖,**只做** code/register/updatePassword。

### P2 · 后台 RBAC 配置链路（mall-admin UMS Role/Resource → gateway）
- **服务/锚点**：`UmsRoleController /role/allocResource`(角色↔资源) / `UmsResourceController`；gateway 读 Redis `auth:pathResourceMap`。
- **真实风险**：**授权变更的正确性 + 动态生效**(改权限→Redis 刷新→无需重启即 403↔200)——现仅测"执行 403",未测**配置侧**。
- **独立讲述**：给角色加某资源→该角色管理员从 403 变 200；撤销→变回 403,验证缓存传播。
- **预估缺陷面**：权限刷新延迟/不生效、越权残留。
- **去重**：不重复 401/403 现象,只测"配置→缓存→执行"闭环。

### P2 · 营销券全生命周期（mall-admin SmsCoupon → portal 领取）
- **服务/锚点**：`SmsCouponController /coupon/create`(count/per_limit/enable_time/end_time/useType) → portal `/member/coupon/add`。
- **真实风险**：券作为**限量+限时资源**——总量领完("已领完")、时间窗外不可领、**并发领取不超发**。
- **独立讲述**：领到 count=0 即拒;未到 enable/已过 end 拒;并发抢领不超总量。
- **去重**：单领 + per_limit 已覆盖(且券此前用 DB 夹具造);此处改为**admin API 造券** + 测**限量/时间/并发**新维度。

### P3 · 退款金额与积分会计（mall-admin OMS return + member 积分）
- **服务/锚点**：`OmsOrderReturnApplyController /returnApply/update/status`(status=2 完成退货 + returnAmount)。
- **真实风险**：**退款金额正确性 + 用券/积分订单退货时的回退会计**——退货**状态机**已测,**金额/积分会计**未测(R6 已暴露取消不退积分,退货路径可能另有问题)。
- **独立讲述**：完成退货后 returnAmount 与原支付一致;若订单用过积分/券,验证是否回退(很可能发现缺陷→新探针 R9?)。
- **去重**：退货 申请→确认/拒绝 已覆盖,只做**钱与积分**维度。

### P3 · 商品聚合与 admin→前台/搜索传播一致性（PmsPortalProduct）
- **服务/锚点**：portal `PmsPortalProductController /product/detail/{id}`(聚合 SKU/属性/阶梯/满减/会员价) ← admin 改 publish/price/stock。
- **真实风险**：**多表聚合正确性 + 改动传播**(admin 下架/改价→前台详情/首页/搜索是否同步、有无延迟/脏读)。
- **独立讲述**：详情聚合各源一致;下架后前台不可见/搜索仍可见(暴露传播不同步)。
- **去重**：浏览"能打开"已覆盖,此处测**聚合一致性 + 传播**。

### P4 · 会员社交（Mongo：关注/收藏/足迹）
- **服务/锚点**：`MemberAttentionController`/`MemberProductCollectionController`/`MemberReadHistoryController`(MongoDB 存储)。
- **真实风险**：**NoSQL 存储链路 + 防重 + 归属**——全新存储介质(此前只 MySQL/Redis/ES)。
- **独立讲述**：关注/收藏幂等(重复不增)、只见自己的、删除生效。
- **去重**：归属/幂等思想同 R2/R1,但**介质与领域全新**,作为 Mongo 样板有讲述价值;业务价值较低,排末位。

## 4. 明确不做（范围外 / 低价值 / 会重复）

- **范围外**：Alipay 真实验签(`AlipayController`)、MinIO/OSS 文件上传(未部署)、UI、性能压测。
- **低价值 CRUD（与已覆盖 admin 商品/订单管理同模式，不重复）**：品牌/分类/属性/退货原因/公司地址/会员等级/菜单/首页广告 等纯增删改查——除非某条带特殊不变量,否则**不做**(同"批量状态/软删"模式已示范)。
- **横切补强（并入各链路负例，不单列）**：非法/越界/注入字符输入校验。

## 5. 建议执行顺序

`P1 秒杀 → P1 注册/认证 → P2 RBAC 配置 → P2 券生命周期 → P3 退款会计 → P3 聚合传播 → P4 Mongo 社交`。
每条沿用样板范式（client/flow/fixture/cases 分层 + 证据驱动 oracle + 数据策略 + Severity/Owner/Issue + 缺陷探针协议），新发现缺陷续编 R9…。
