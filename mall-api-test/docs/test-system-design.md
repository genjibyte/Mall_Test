# 测试工程体系设计：从个人项目趋近大厂落地体系

> 不堆技术栈，用**工程难题**驱动；把个人项目做成"更难但可控"的测试体系，趋近大厂方法论。
> **诚实边界**：有真实微服务 SUT(mall-swarm) + 源码级真实缺陷，但**无真实用户流量、无生产缺陷库、无历史回归资产**。故下文标"尝试/模拟"者，价值在**工程实践与方法论可迁移**，用 STAR(§4) 讲清"解决什么 / 怎么落地 / 产出 / 效率"，而非罗列工具。
>
> **三条边界（贯穿全程，超出即不做）**：
> - **诚实边界**：受控模拟。逃逸率 / ROI / 线上 MTTR 等无资产支撑的指标只能**标注为模拟**，不夸大为生产数据。
> - **范围边界**：**不改 SUT(mall-swarm)**；真实支付验签 / OSS-MinIO / UI / 性能压测 明确范围外。
> - **可控边界**：「更难但可控」——每个新难题必须本地**可控注入、可复现**(如 `docker stop` 注依赖故障)，不引入不可控外部依赖或不可重复的复杂度。

## 1. 大厂测试体系维度全景（补全你的表述）

你点名了 数据隔离 / 独立 / 并发 / 微服务 / 中间件。一个成熟测试工程体系还包含以下维度——下表是**全景 × 现状 × 趋近目标**：

| # | 维度 | 大厂典型做法 | 本项目现状 | 趋近目标（尝试） |
|---|---|---|---|---|
| 1 | 测试策略/分层 | 金字塔(单元>集成>e2e)+契约;风险驱动选择 | API 集成层 60 用例、风险驱动 | 加契约层；按 diff 做**用例影响分析**(只跑受影响) |
| 2 | 环境管理 | 环境即代码、按 PR 临时环境、与生产对等 | 固定本地隔离栈 + 恢复 SOP | Testcontainers 全新库/容器、CI 临时环境 |
| 3 | 测试数据 | 工厂/Builder、租户隔离、合成数据、泄漏检测 | 三策略+双隔离夹具+完整性守卫+回收 | 数据工厂、并行分区、golden dataset |
| 4 | 并发/可靠性 | flaky 治理(隔离/重试/评分)、竞态系统化 | 超卖探针 + MQ 真实异步 | 幂等矩阵、flaky 看板、并行安全 |
| 5 | **微服务** | 消费者驱动契约、服务虚拟化、网关韧性 | 经网关端到端、跨服务建品→ES | 契约测试、WireMock 桩、熔断/重试/超时 |
| 6 | **中间件** | 依赖**故障注入**(混沌)、一致性/事务边界 | MySQL/Redis/MQ/ES 功能链路 | 故障注入:依赖宕机下的降级行为 |
| 7 | 质量门禁/CI | 多阶段流水线、覆盖率门槛、**变异测试** | slow 标签 + 缺陷协议 + @Disabled 门禁 | CI(快/慢/nightly)、PIT 变异测试度量"有效性"、趋势看板 |
| 8 | 可观测/报告 | 历史趋势、flaky 看板、缺陷追溯、MTTR | Allure 环境/分类/Severity/Owner/Issue | Allure history 趋势、flaky 率、缺陷链 issue |
| 9 | 安全/契约 | authz 矩阵、注入、OpenAPI 契约校验 | 鉴权执行(401/403/越权探针) | 越权矩阵、注入横切、响应 Schema 校验 |
| 10 | 缺陷/效能度量 | 检出率/逃逸率/ROI/反馈时间 | 5 真实缺陷固化为探针 | 反馈时间/flaky 率/缺陷密度 度量化 |
| 11 | 左移 | 需求即用例、spec-by-example | 证据驱动 oracle(源码级) | 从需求/契约生成用例 |

> 加粗(微服务/中间件/变异测试)是与"个人项目"差距最大、最值得"尝试"的三块。

## 2. 重点探索设计（你点名的几项，给可落地方案）

### 2.1 数据隔离与独立（已有基础 → 再难一档）
- 现状：取消还原 / 快照还原 / **双隔离夹具**(专用会员+SKU) + **库存完整性守卫**(漂移即门禁失败) + 维护回收。
- 尝试：① **数据工厂/Builder**(流式构造 order/member/coupon，替散落夹具)；② **并行分区**(每线程专属数据段，解除串行)；③ **Testcontainers** 全新库(每次 run 干净，彻底独立)；④ 泄漏检测扩到 订单/券/积分 维度。
- **难题→价值**：HTTP 级测试**无法事务回滚**(SUT 自进程提交)——隔离只能靠"专属数据 + 守卫 + 回收"，这是与单元测试**根本不同**的工程难点，正是可讲之处。

### 2.2 并发与可靠性
- 尝试：① **幂等矩阵**(对所有 mutation 列"重复提交"用例)；② **竞态系统化**(领券/秒杀限购的 read-modify-write 竞态)；③ **flaky 治理**(quarantine 标签 + 重试策略 + 失败率评分)；④ 完成隔离后**开并行**。

### 2.3 微服务（当前最大方法论缺口）
- 尝试：① **消费者驱动契约**(Spring Cloud Contract：gateway↔服务、服务↔服务 Feign 的请求/响应契约，防"接口悄悄改坏")；② **服务虚拟化**(WireMock——你机上已有 `order-test-wiremock` 栈，把被测服务的下游 mock 掉，隔离单服务测)；③ **网关韧性**(对某服务注入延迟/错误，验证熔断/重试/超时/降级)。

### 2.4 中间件（从"用"到"压"）
- 尝试：**依赖故障注入(轻量混沌)**，且**可控**(`docker stop` 即注入)：
  - Redis 宕 → 会员缓存读取是否降级 / 会话失效行为？
  - RabbitMQ 宕 → 下单是否仍成功(超时取消消息发不出会怎样)？
  - MySQL 连接池打满 → 超时与排队行为？
  - ES 宕 → 搜索降级还是 500？（本会话 ES cgroup 事故已是一次真实"ES 不可用"演练）
- **价值**："依赖坏了会怎样"是大厂韧性测试核心，本项目可控地模拟。

### 2.5 质量效能（证明"测试抓得住"）
- 尝试：① **变异测试**(PIT：故意改源码看测试是否抓到 → 度量**有效性**而非覆盖率)；② **CI 多阶段**(PR 快线 59 用例 / nightly `-Pslow` + 变异 + 全链路)；③ **趋势看板**(Allure history：通过率/flaky/时长趋势)。

## 3. 落地路线（分阶段，每阶段啃一个"难题"，可独立讲）

| 阶段 | 难题(一句话) | 动作 | 产出/可讲点 |
|---|---|---|---|
| S1 | 共享态 → 可并行 | 数据工厂 + 分区 + 开并行 | 套件时长↓、独立性↑ |
| S2 | 微服务接口悄悄改坏 | 消费者驱动契约 | 契约破坏即红 |
| S3 | 依赖故障下的降级 | 中间件故障注入(Redis/MQ/DB/ES) | 韧性边界用例 |
| S4 | 测试"抓不抓得住" | 变异测试 + CI 门禁 | 有效性量化 |
| S5 | 隔离被测服务 | WireMock 服务虚拟化 | 单服务可独立测 |

## 4. STAR 叙事库（怎么讲这个项目：挑难点、说方案、给产出）

> 讲法：不说"我用了 JUnit/RestAssured/Allure"，而说"遇到 X 难题，怎么解，产出什么，效率如何"。以下均为**本项目真实经历**。

**STAR-1 · 测试数据漂移与隔离**
- **S** 多用例共享种子会员+SKU，HTTP 级测试无法事务回滚，跑多了负库存/订单膨胀，无法重复、无法并行。
- **T** 让套件可重复、不漂移、为并行铺路。
- **A** 三层数据策略(取消还原/快照还原/双隔离夹具) + **常驻库存完整性守卫**(负库存/超锁即门禁失败) + 维护回收工具。
- **R** 60 用例可重复全绿；一次真实漂移(负库存)被守卫自动检出并修复；隔离夹具就位。

**STAR-2 · 环境可复现（cgroup v2 事故）**
- **S** WSL 内核升级引入 cgroup v2，ES 7.17.3 自带 JDK 崩溃，Docker 三次卡死，3 种标准修法无效。
- **T** 恢复并固化"可复现"环境。
- **A** 定位根因(JDK 的 cgroup v2 解析 bug) → 升 ES 7.17.18 换新 JDK + 匹配 IK → 记录无效修法防重复 + 写恢复 SOP。
- **R** 全 60 绿闭环；环境恢复手册沉淀，下次同类问题分钟级解决。

**STAR-3 · 真实异步最终一致（MQ 超时）**
- **S** 订单超时走 RabbitMQ TTL+DLX，真实延迟路径未测，且有"按消息 TTL 队头阻塞"陷阱。
- **T** 验证真实异步链路而非仅同步取消。
- **A** 降 overtime + 清 ttl 队列消队头阻塞 + Awaitility 轮询 + `@Tag(slow)` 隔离不拖快线。
- **R** 真实 ~60s 异步关单+释放库存验证通过，nightly 可跑。

**STAR-4 · 缺陷协议（留证不阻断）**
- **S** 测试中挖到 5 个真实缺陷(非幂等支付/越权支付/并发超卖/积分不退/关单泄漏)，但不能让 CI 一直红。
- **T** 既留缺陷证据，又不阻断门禁，修复后自动转回归。
- **A** `@KnownDefect` 探针——按"正确行为"断言 → 跑一次"失败=暴露缺陷"留证 → 默认 `@Disabled` → `@Issue(Rn)` 可追溯。
- **R** 5 缺陷固化为守护；修复删注解即转回归用例，零额外成本。

**STAR-5 · 证据驱动 oracle（抗数据变化）**
- **S** 金额/促销预期易写成魔法数，数据一改就脆。
- **T** 预期可追溯、抗数据漂移。
- **A** 复刻源码算法(BigDecimal 同口径)从 DB 配置现算 + 断差值而非绝对值。
- **R** 首次即与应用对齐；边界/等价类数据驱动 6 用例一次通过。

## 5. 效率与产出（讲"效率如何"用数字）

- **反馈时间**：默认快线 59 用例数十秒；慢用例(真实 MQ)`@slow` 隔离走 nightly，不拖 PR。
- **缺陷产出**：5 个源码级真实缺陷固化为探针(R1/R2/R4/R6/R8)，修复即转守护。
- **可复现**：漂移自动检出+一键修复；环境恢复 SOP。
- **可追溯**：每用例 Epic/Feature/Story/Severity/Owner/Issue，报告按严重度/负责人/缺陷过滤。
- **维护成本**：分层复用(client/flow/fixture)，`OrderFlow.placeOrder` 消重；新增用例有检查清单。
- **诚实差距**：无真实用户/生产缺陷库/历史回归资产——"逃逸率/ROI"等只能**模拟**，价值在方法论可迁移。

## 6. 难题目录（挑战清单 · 标注边界）

> 每条标 `[已解·可讲]`（即 STAR 素材）/ `[尝试·可控]`（即 §3 路线）/ `[边界·不做]`。

**`[已解·可讲]` —— 本项目真实啃过，可直接讲：**

| 难题 | 症结(一句话) | 解法 |
|---|---|---|
| 数据漂移/隔离 | HTTP 级测试无法事务回滚 | 三策略 + 双隔离 + 完整性守卫 + 回收 |
| 环境可复现 | WSL cgroup v2 崩 ES、Docker 卡死 | 升 ES 7.17.18 换 JDK + 恢复 SOP |
| 真实异步一致 | MQ 按消息 TTL 队头阻塞 | 清队列 + Awaitility + slow 隔离 |
| 缺陷留证不阻断 | 真缺陷不能让门禁红 | @KnownDefect 探针 + @Issue 追溯 |
| oracle 抗数据变化 | 金额魔法数易脆 | 复刻源码算法、断差值 |
| 断言契约反直觉 | 业务失败返回 HTTP 200 + code 500 | 统一先断 body.code |
| 双账号鉴权 | Sa-Token 两套 token 命名空间 | 双 token 工厂 |
| 缓存一致性 | 改积分后 Redis 缓存读旧值 | 读前 invalidateCache |
| 确定性 403 | 种子受限账号密码未知 | 复制 admin 密码哈希使其可登 |
| 测试自身可信 | 测试绿 ≠ 测试有效 | 探针"解禁即失败"反证（+ 变异测试） |
| 漂移 vs 缺陷归因 | 失败难判是 bug 还是脏数据 | 完整性守卫 + Allure categories 分类 |
| 慢用例不拖快线 | 真实 MQ 用例 ~60s | @slow 排除 + `-Pslow` nightly |

**`[尝试·可控]` —— 受控模拟，非生产（详见 §2/§3）：**
- 中间件故障注入：`docker stop` Redis/MQ/DB/ES → 验降级（可控、可复现）。
- 契约破坏检测：消费者驱动契约（模拟"接口悄悄改坏"）。
- 测试有效性：变异测试 PIT（度量"抓得住"而非覆盖率）。
- 并行安全：数据分区后开并行。
- 服务虚拟化：WireMock 隔离单服务（机上已有 order-test-wiremock 栈）。

**`[边界·不做]`：**
- 真实用户流量回放 / 生产缺陷库 / 线上指标——无资产，不伪造成真实。
- 真实支付验签、OSS/MinIO、UI、性能压测、**改 SUT 源码**——范围外。
