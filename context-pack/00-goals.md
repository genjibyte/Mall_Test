# 00 · 目标（Goals）

## 一句话目标

为 mall-swarm 微服务电商系统建立一套**可重复、可回归、可进 CI** 的**接口自动化测试框架**，
覆盖核心业务链路，验证接口契约与业务正确性，并能稳定暴露既有缺陷。

## 背景与定位

- 角色：资深测试开发，独立负责该测试框架。
- 被测系统：mall-swarm（已在本地全栈部署，见 [04-runtime-environment.md](04-runtime-environment.md)）。
- 测试形态：以**黑盒 API 测试**（经网关 `:8201`）为主，辅以**直连 MySQL 的灰盒校验**（库存/订单/券/积分状态）。
- 本框架服务于"业务链路测试"，是后续编码的目标产物。

## 分层目标

### G1 接口契约正确性
每个被测接口的请求/响应符合契约：统一响应体 `{code,message,data}`、`code` 语义（200/401/403/404/500）、`data` 结构与字段类型。契约见 [02-code-context.md](02-code-context.md)。

### G2 业务正确性（核心）
验证关键链路的业务规则与状态流转，至少覆盖 5 条链路（见 [01-scope-boundaries.md](01-scope-boundaries.md)）：
- **下单主链路**：购物车→确认单→下单→支付→收货；金额计算、库存锁定/扣减、券核销、积分扣减/赠送。
- **认证与 RBAC**：双账号登录、token 鉴权、越权拦截。
- **订单超时自动取消**：MQ 延迟 → 取消 → 库存/券/积分回滚（最终一致）。
- **商品上架→搜索同步**：DB→ES 导入与检索。
- **优惠券营销**：建券→领取→下单抵扣→退单回滚。

### G3 缺陷暴露
对已知源码级缺陷（[06-historical-badcases.md](06-historical-badcases.md)，如 `paySuccess` 非幂等、越权支付、超卖）编写**会失败的断言**，使框架成为回归/防劣化的探针，而非只跑 happy path。

### G4 可回归 / 可自动化
- 用例可重复执行、相互隔离、结果确定（含异步链路的"轮询直到"原语）。
- 可一键运行、产出结构化报告、可接入 CI 门禁（见 [05-quality-gates.md](05-quality-gates.md)）。

## 成功判据（Definition of Success）

| 判据 | 衡量 |
|---|---|
| P0 链路覆盖 | 下单主链路 + 认证 happy path 与关键负例 100% 有用例并通过 |
| 数据自洽 | 用例自带前置数据准备与后置清理，重复跑结果一致 |
| 缺陷可复现 | R1/R2/R4 类缺陷用例能稳定复现（或在修复后转为守护） |
| 断言正确 | 全部断言遵循"先 `body.code` 后 `data`"，金额 BigDecimal 精确 |
| 可交付 | 框架有 README、可一键跑、报告可读、新成员按本包能接手 |

## 非目标（详见 [01-scope-boundaries.md](01-scope-boundaries.md)）

不做前端 UI 自动化、不做源码单元测试、不做性能/压测（另立项）、不修改 mall-swarm 源码、不覆盖 mall-monitor/mall-demo 与未部署的 MinIO/OSS/支付宝真实集成。
