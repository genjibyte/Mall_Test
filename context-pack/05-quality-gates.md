# 05 · 质量门禁（Quality Gates）

> 测试用例与框架必须满足的硬规则。违反则视为不合格，不得合入。

## QG1 断言规则（最高优先）

1. **先断 `body.code`，再断 `data`**。`code` 是业务码（200/401/403/404/500），与 HTTP 状态**解耦**——业务失败常见 `HTTP 200 + body.code=500`。只判 HTTP status 视为错误断言。
2. **正向**：`code==200` 且 `message=="操作成功"`（或接口自定义成功语）且 `data` 结构/值符合预期。
3. **负向**：精确断言期望的 `code` 与 `message`（如库存不足→`code 500` + `"库存不足，无法下单"`；未登录→`code 401`；越权→`code 403`）。
4. **金额**：一律 `BigDecimal`，用 `compareTo` 或对齐 scale 比较，禁止 double/`equals`。
5. **副作用**：凡接口产生 DB/缓存/MQ 副作用的，必须**查库复核**（库存、订单 status、券 use_status、积分），不能只信响应。

## QG2 测试隔离与可重复

1. **数据独立**：用例自带前置准备（建会员数据/加购/建券…）；优先用独立/新建数据，避免共享种子被污染。
2. **后置清理/复原**：改了共享状态（如临时改 `oms_order_setting.normal_order_overtime`、扣减了库存）必须在 teardown 复原或基于事务/独立数据。
3. **无顺序依赖**：用例之间不得隐式依赖执行顺序或彼此产生的数据。
4. **库存漂移防护**：下单类用例会改 `lock_stock`，必须取消/复原，否则污染后续用例的 `realStock`。

## QG3 异步与确定性

1. 订单超时取消、ES 导入等为**最终一致**，断言用"**轮询直到满足/超时**"原语，不得用裸 `sleep` 凑时间。
2. ES 写入有 ~1s 刷新延迟；搜索断言需容忍刷新或显式等待可见。
3. 时间相关用例（超时）通过**改配置缩短窗口**（normal_order_overtime→1min）+ 轮询，跑完复原。

## QG4 覆盖目标（分层）

| 层级 | 要求 |
|---|---|
| P0 | 下单 happy path、认证 happy path + 401/403 负例：**必须有且通过** |
| P1 | 下单边界/金额（券·积分·促销）、取消回滚、搜索、优惠券领取抵扣 |
| P2 | 已知缺陷探针（幂等/越权/超卖）、管理端订单流转 |
| — | 每条链路至少 1 正向 + 关键负向；金额类用例数据驱动覆盖多组合 |

## QG5 缺陷处理协议

- 命中 [06-historical-badcases.md](06-historical-badcases.md) 的已知缺陷：用例**按"正确行为"断言**（预期失败即真实缺陷），并打标 `@KnownDefect(Rxx)`，进报告但不阻断门禁（避免被既有 bug 卡死 CI）。
- 新发现疑似缺陷：记录到 06 文件（症状/证据/复现），不擅自当成"环境问题"绕过。

## QG6 命名与结构约定

- 用例命名体现 链路_场景_预期，如 `order_generate_outOfStock_shouldFail`。
- 分层：`接口客户端层`（封装网关调用+token）/`业务流程层`（链路编排）/`数据夹具层`（DB 准备校验）/`用例层`。
- 断言信息可读：失败时输出实际 `code/message/data` 与关键 DB 状态。

## QG7 Definition of Done（套件级）

一个测试套件"完成"需同时满足：
- [ ] 能一键运行（含环境自检：Nacos 5 服务健康 + 登录冒烟）。
- [ ] P0 用例全绿；P1 主要场景有覆盖。
- [ ] 无顺序依赖、可重复跑两遍结果一致（数据自洽）。
- [ ] 断言遵循 QG1；金额 BigDecimal；副作用有 DB 复核。
- [ ] 产出结构化报告（通过/失败/已知缺陷分类）。
- [ ] 不修改 mall-swarm 源码；测试产物在仓库外。

## QG8 CI 门禁（接入时）

1. 前置：起基础设施 + 服务 + 健康自检通过才进入用例阶段。
2. 通过条件：P0 全绿、无 flaky（重试仍稳定）、报告生成。
3. `@KnownDefect` 用例不计入阻断，但需在报告单列，便于"修复后转守护"。
